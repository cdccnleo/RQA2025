# 特征数据存储方案优化计划

## 问题描述

当前特征选择历史数据存储 (`data/feature_selection_history.json`) 仅使用文件系统，与整个量化交易系统采用的 PostgreSQL + 文件系统双存储方案不符。

## 优化目标

将特征选择历史数据存储从单一文件系统升级为 PostgreSQL + 文件系统双存储方案，与系统整体架构保持一致。

## 当前状态

### 现有存储方案
- **文件**: `data/feature_selection_history.json`
- **格式**: JSON 数组
- **管理类**: `FeatureSelectorHistoryManager`

### 系统整体存储方案
- **主存储**: PostgreSQL 数据库
- **降级存储**: 文件系统
- **同步机制**: PostgreSQL 优先，文件系统备份

## 优化内容

### 1. 数据库表设计

创建 `feature_selection_history` 表：

```sql
CREATE TABLE IF NOT EXISTS feature_selection_history (
    id SERIAL PRIMARY KEY,
    selection_id VARCHAR(100) UNIQUE NOT NULL,
    task_id VARCHAR(100) NOT NULL,
    timestamp TIMESTAMP NOT NULL,
    input_features JSONB,
    input_feature_count INTEGER DEFAULT 0,
    selection_method VARCHAR(50),
    selection_params JSONB,
    selected_features JSONB,
    selected_feature_count INTEGER DEFAULT 0,
    selection_ratio FLOAT DEFAULT 0.0,
    evaluation_metrics JSONB,
    processing_time FLOAT DEFAULT 0.0,
    notes TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_feature_selection_task_id ON feature_selection_history(task_id);
CREATE INDEX idx_feature_selection_timestamp ON feature_selection_history(timestamp);
CREATE INDEX idx_feature_selection_method ON feature_selection_history(selection_method);
```

### 2. 代码优化

#### 2.1 添加 PostgreSQL 存储支持

在 `FeatureSelectorHistoryManager` 中添加：
- `_save_to_postgresql()` 方法
- `_load_from_postgresql()` 方法
- `_sync_to_filesystem()` 方法（降级备份）

#### 2.2 修改数据加载逻辑

优先从 PostgreSQL 加载，如果失败则从文件系统加载并同步到 PostgreSQL。

#### 2.3 修改数据保存逻辑

同时保存到 PostgreSQL 和文件系统，确保数据一致性。

### 6. 预期结果

1. ✅ 特征选择历史数据存储与系统整体架构一致
2. ✅ 支持 PostgreSQL + 文件系统双存储
3. ✅ 数据查询性能提升（利用数据库索引）
4. ✅ 数据可靠性提升（双备份机制）
5. ✅ 向后兼容（文件系统作为降级方案）

### 7. 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 数据迁移失败 | 高 | 保留原始 JSON 文件，可回滚 |
| PostgreSQL 连接失败 | 中 | 降级到文件系统，不影响功能 |
| 性能下降 | 低 | 添加数据库索引，优化查询 |

3. 测试双存储方案是否正常工作
4. 验证降级机制是否有效
