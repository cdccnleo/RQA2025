# 优化策略回测分析页面仪表盘加载速度

## 问题分析

通过分析代码和日志，我发现仪表盘加载缓慢的主要原因：

1. **回测引擎执行时间过长**：
   - 多次尝试不同的调用方法（4种不同的调用方式）
   - 每次调用都生成完整的模拟数据
   - 数据处理和转换开销大

2. **数据处理开销**：
   - 生成模拟数据时创建完整的DataFrame
   - 结果转换需要多次调用`tolist()`和`to_dict()`
   - 处理`BacktestResult`对象的复杂逻辑

3. **持久化存储操作**：
   - 回测完成后立即保存到文件系统和PostgreSQL
   - I/O操作阻塞主线程

4. **事件发布和WebSocket广播**：
   - 多个事件发布和WebSocket广播操作
   - 增加了响应时间

5. **ML层服务集成**：
   - 每次都尝试获取MLCore和模型管理器实例
   - 增加了初始化时间

## 解决方案

### 1. 优化回测引擎调用
- **减少尝试次数**：只尝试最可能成功的调用方式
- **缓存模拟数据**：避免重复生成相同的模拟数据
- **异步执行**：将回测引擎执行移到后台线程

### 2. 优化数据处理
- **延迟数据转换**：只在需要时进行数据转换
- **优化结果构建**：简化返回字典的构建逻辑
- **减少数据量**：对大型数据集进行分页或采样

### 3. 优化持久化存储
- **异步保存**：将持久化操作移到后台线程
- **批量保存**：合并多个保存操作
- **缓存结果**：避免重复加载相同的结果

### 4. 优化事件和WebSocket操作
- **减少事件发布**：只发布必要的事件
- **批量广播**：合并WebSocket广播消息
- **异步广播**：将广播操作移到后台线程

### 5. 优化ML层服务集成
- **延迟初始化**：只在需要时初始化ML层服务
- **缓存实例**：避免重复创建相同的服务实例

## 具体实现步骤

1. **修改`run_backtest`函数**：
   - 引入异步执行机制
   - 优化ML层服务初始化逻辑
   - 添加结果缓存

2. **修改`_execute_backtest`函数**：
   - 减少尝试次数，优先使用最可能成功的方法
   - 优化模拟数据生成逻辑
   - 简化结果处理和转换

3. **修改持久化操作**：
   - 将`save_backtest_result`改为异步函数
   - 在后台线程执行保存操作

4. **优化事件和WebSocket操作**：
   - 减少事件发布频率
   - 批量处理WebSocket广播

5. **添加性能监控**：
   - 添加执行时间统计
   - 记录关键操作的耗时

## 预期效果

- **响应时间减少**：仪表盘加载时间从可能的数十秒减少到数秒
- **用户体验改善**：页面加载更加流畅，不再长时间卡顿
- **系统资源优化**：减少CPU和内存使用
- **可扩展性提升**：更好地处理并发请求
