# 数据采集监控任务管理功能迭代开发计划

## 目标

按照改进建议优先级，对 `data-collection-monitor.html` 页面任务管理标签进行迭代完善，提升历史数据采集调度与任务管理的精细化程度。

## 迭代阶段规划

### 第一阶段：高优先级功能实现（预计 2-3 天）

#### 1.1 实现最近完成任务列表

**前端开发** (`data-collection-monitor.html`):
- 修改第 712-735 行的 completed-tasks-table 区域
- 添加任务数据加载函数 `loadCompletedTasks()`
- 实现任务行渲染，显示：任务ID、标的、记录数、耗时、完成时间
- 添加分页或滚动加载（如果任务量大）

**后端开发** (`src/gateway/web/datasource_routes.py`):
- 新增 API: `GET /api/v1/data/scheduler/tasks/completed`
- 从统一调度器获取已完成任务历史
- 支持分页参数：limit, offset
- 返回字段：task_id, source_id, records_count, duration_ms, completed_at, status

**数据存储**:
- 利用统一调度器已有的 completed_tasks 统计
- 扩展任务元数据存储，记录完成详情

**验收标准**:
- [ ] 页面加载时自动显示最近完成的任务列表
- [ ] 表格包含任务ID、标的、记录数、耗时、完成时间列
- [ ] 数据实时更新（每30秒刷新）
- [ ] 无数据时显示友好提示

#### 1.2 实现任务详情查看

**前端开发**:
- 添加任务详情弹窗/抽屉组件
- 实现 `showTaskDetail(taskId)` 函数
- 显示内容：
  - 基本信息：任务ID、数据源、提交时间、完成时间
  - 执行状态：成功/失败、错误信息（如果失败）
  - 采集统计：记录数、数据大小、耗时
  - 执行日志：关键步骤日志

**后端开发**:
- 新增 API: `GET /api/v1/data/scheduler/tasks/{task_id}`
- 返回单个任务的完整信息
- 从统一调度器获取任务执行详情

**验收标准**:
- [ ] 点击任务行可打开详情弹窗
- [ ] 弹窗显示完整的任务执行信息
- [ ] 支持关闭弹窗
- [ ] 加载失败时显示错误提示

#### 1.3 完善配置持久化

**前端开发**:
- 实现 `loadCollectionPeriodConfig()` 函数
  - 调用 API 获取当前配置
  - 设置采集期类型选择框的值
- 实现 `updateCollectionPeriodConfig()` 函数
  - 获取选择框值
  - 调用 API 保存配置
  - 显示保存成功/失败提示
- 实现 `updateLatencyThresholds()` 函数
  - 获取阈值输入框值
  - 调用 API 保存阈值配置

**后端开发**:
- 新增配置存储机制（文件或数据库）
- 新增 API: `GET /api/v1/data/scheduler/config`
  - 返回采集期类型、延迟阈值等配置
- 新增 API: `POST /api/v1/data/scheduler/config`
  - 保存配置参数
  - 支持字段：collection_period_type, latency_warning_threshold, latency_critical_threshold

**配置存储方案**:
- 方案A：JSON文件存储（简单快速）
  - 路径：`config/scheduler_config.json`
- 方案B：数据库存储（推荐，便于扩展）
  - 使用现有 PostgreSQL
  - 表名：`scheduler_config`

**验收标准**:
- [ ] 页面加载时自动加载保存的配置
- [ ] 修改配置后点击保存，配置持久化
- [ ] 刷新页面后配置保持不变
- [ ] 保存成功/失败有明确提示

### 第二阶段：中优先级功能实现（预计 3-4 天）

#### 2.1 添加单个任务控制

**前端开发**:
- 在任务列表添加操作列
- 添加操作按钮：
  - 暂停按钮（运行中任务显示）
  - 恢复按钮（暂停任务显示）
  - 取消按钮（待处理任务显示）
  - 重试按钮（失败任务显示）
- 实现对应操作函数：
  - `pauseTask(taskId)`
  - `resumeTask(taskId)`
  - `cancelTask(taskId)`
  - `retryTask(taskId)`

**后端开发**:
- 扩展统一调度器 API:
  - `POST /api/v1/data/scheduler/tasks/{task_id}/pause`
  - `POST /api/v1/data/scheduler/tasks/{task_id}/resume`
  - `POST /api/v1/data/scheduler/tasks/{task_id}/cancel`
  - `POST /api/v1/data/scheduler/tasks/{task_id}/retry`
- 在统一调度器中实现任务状态管理
- 支持任务优先级调整 API:
  - `PUT /api/v1/data/scheduler/tasks/{task_id}/priority`
  - 参数：priority (high/normal/low)

**验收标准**:
- [ ] 可以对单个运行中任务执行暂停
- [ ] 可以对单个暂停任务执行恢复
- [ ] 可以对单个待处理任务执行取消
- [ ] 可以对单个失败任务执行重试
- [ ] 可以调整任务优先级
- [ ] 操作后有明确反馈提示

#### 2.2 增强任务监控

**WebSocket 实时更新**:
- 前端：完善 `handleWebSocketMessage()` 函数
  - 处理 task_progress 消息，更新任务进度条
  - 处理 task_completed 消息，刷新任务列表
  - 处理 task_failed 消息，显示告警
- 后端：在统一调度器中发送 WebSocket 消息
  - 任务进度更新时广播
  - 任务完成/失败时广播

**任务执行日志查看**:
- 前端：在任务详情弹窗添加日志标签页
  - 显示实时日志流
  - 支持日志级别筛选
- 后端：新增 API `GET /api/v1/data/scheduler/tasks/{task_id}/logs`
  - 返回任务执行日志
  - 支持 SSE (Server-Sent Events) 实时推送

**失败任务重试机制**:
- 后端：在统一调度器中实现自动重试
  - 配置最大重试次数（默认3次）
  - 实现指数退避策略
  - 记录重试历史

**验收标准**:
- [ ] 任务执行时实时显示进度
- [ ] 任务完成自动刷新列表
- [ ] 可以查看任务执行日志
- [ ] 失败任务自动重试（可配置）
- [ ] 重试历史可查看

### 第三阶段：低优先级功能实现（预计 4-5 天）

#### 3.1 任务调度策略优化

**支持更多调度策略**:
- 时间窗口策略：只在指定时间段执行
- 依赖关系策略：任务A完成后自动触发任务B
- 批量策略：合并多个小任务批量执行

**任务分组和批处理**:
- 支持按数据源类型分组
- 批量提交任务
- 批量操作（批量暂停/恢复/取消）

**智能负载均衡**:
- 根据工作进程负载动态分配任务
- 根据数据源响应时间调整并发度
- 自适应采集频率调整

#### 3.2 历史数据分析

**采集趋势分析图表**:
- 使用 Chart.js 绘制趋势图
- 显示指标：
  - 每日采集量趋势
  - 成功率趋势
  - 平均耗时趋势
  - 数据源健康度趋势

**性能瓶颈识别**:
- 自动识别慢数据源
- 识别高频失败数据源
- 生成性能报告

**预测性维护建议**:
- 基于历史数据预测数据源故障
- 提前告警可能的问题
- 给出优化建议

## 技术实现细节

### API 设计规范

```
# 任务列表
GET /api/v1/data/scheduler/tasks?status=completed&limit=20&offset=0
Response: {
  "tasks": [...],
  "total": 100,
  "limit": 20,
  "offset": 0
}

# 单个任务详情
GET /api/v1/data/scheduler/tasks/{task_id}
Response: {
  "task_id": "...",
  "source_id": "...",
  "status": "completed|failed|running|pending",
  "submitted_at": "...",
  "started_at": "...",
  "completed_at": "...",
  "records_count": 1000,
  "duration_ms": 5000,
  "error_message": "...",
  "logs": [...]
}

# 任务控制
POST /api/v1/data/scheduler/tasks/{task_id}/pause
POST /api/v1/data/scheduler/tasks/{task_id}/resume
POST /api/v1/data/scheduler/tasks/{task_id}/cancel
POST /api/v1/data/scheduler/tasks/{task_id}/retry

# 配置管理
GET /api/v1/data/scheduler/config
POST /api/v1/data/scheduler/config
Body: {
  "collection_period_type": "quarterly",
  "latency_warning_threshold": 5000,
  "latency_critical_threshold": 10000
}
```

### 数据库表设计（如果需要）

```sql
-- 调度器配置表
CREATE TABLE scheduler_config (
    id SERIAL PRIMARY KEY,
    key VARCHAR(100) UNIQUE NOT NULL,
    value JSONB NOT NULL,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 任务历史表（如果需要持久化任务历史）
CREATE TABLE task_history (
    id SERIAL PRIMARY KEY,
    task_id VARCHAR(100) UNIQUE NOT NULL,
    source_id VARCHAR(100) NOT NULL,
    status VARCHAR(20) NOT NULL,
    submitted_at TIMESTAMP,
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    records_count INTEGER DEFAULT 0,
    duration_ms INTEGER,
    error_message TEXT,
    metadata JSONB
);
```

### 前端组件设计

```javascript
// 任务管理器类
class TaskManager {
  async loadCompletedTasks(limit = 20, offset = 0);
  async getTaskDetail(taskId);
  async pauseTask(taskId);
  async resumeTask(taskId);
  async cancelTask(taskId);
  async retryTask(taskId);
  async updateTaskPriority(taskId, priority);
  subscribeToTaskUpdates(callback);
}

// 配置管理器类
class ConfigManager {
  async loadConfig();
  async saveConfig(config);
  async updateCollectionPeriod(type);
  async updateLatencyThresholds(warning, critical);
}
```

## 风险评估与应对

### 技术风险

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 统一调度器API不兼容 | 高 | 提前与后端团队确认API设计 |
| WebSocket连接不稳定 | 中 | 实现自动重连机制，降级到轮询 |
| 大量任务数据性能问题 | 中 | 实现分页加载，虚拟滚动 |
| 配置存储方案变更 | 低 | 使用抽象层封装配置操作 |

### 进度风险

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 需求变更 | 中 | 每个阶段结束后进行评审 |

## 备注

1. **与现有功能兼容性**: 所有新功能必须与现有功能兼容，不能破坏已有功能
2. **向后兼容性**: API设计考虑向后兼容，便于后续扩展
3. **性能考虑**: 大量数据场景下需要分页和懒加载
4. **用户体验**: 操作反馈要及时明确，加载状态要可见
5. **错误处理**: 所有API调用需要完善的错误处理和用户提示
