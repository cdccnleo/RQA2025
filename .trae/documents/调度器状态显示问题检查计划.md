# 调度器状态显示问题检查计划

## 问题描述

特征工程监控页面中特征任务调度器状态仪表盘显示：
- 调度器状态：已停止
- 活跃工作节点：0

需要系统化检查这些显示是否正确。

## 检查目标

1. 确认调度器状态显示逻辑是否正确
2. 确认活跃工作节点计算逻辑是否正确
3. 确认任务数据是否正确加载
4. 确认前后端数据交互是否正常

## 检查步骤

### 步骤1: 检查后端任务数据
- 检查 `feature_task_persistence.list_feature_tasks()` 返回的数据
- 确认任务状态字段（pending/running/completed/failed）
- 确认是否有 running 状态的任务

### 步骤2: 检查后端调度器状态计算逻辑
- 检查 `get_scheduler_status()` 函数实现
- 确认 `is_running` 计算逻辑（running_count > 0）
- 确认 `active_workers` 计算逻辑

### 步骤3: 检查前端状态显示逻辑
- 检查 `updateSchedulerStatus()` 函数实现
- 确认状态文本显示逻辑（运行中/已停止）
- 确认状态指示器颜色逻辑（绿色/红色）

### 步骤4: 检查API数据交互
- 检查 `/features/engineering/scheduler/status` 接口返回数据
- 确认数据格式是否匹配
- 检查网络请求是否正常

### 步骤5: 检查任务调度器实际运行状态
- 检查是否有实际运行的调度器进程
- 检查调度器是否正确管理任务状态
- 检查任务状态是否正确持久化

### 步骤6: 验证测试
- 创建测试任务，观察状态变化
- 启动调度器，观察状态更新
- 验证数据显示是否正确

## 预期结果

1. 如果有 running 状态的任务，调度器状态应显示"运行中"
2. 活跃工作节点应显示 running 状态的任务数量
3. 如果没有任务或所有任务都已完成，显示"已停止"是正确的

## 可能的原因

1. 确实没有 running 状态的任务（正常情况）
2. 任务状态没有正确持久化
3. 前端数据解析错误
4. 后端数据计算逻辑错误
5. API 接口返回数据格式不匹配

## 验证方法

1. 查看容器日志确认任务状态
2. 直接查询数据库确认任务数据
3. 使用浏览器开发者工具检查 API 响应
4. 手动创建测试任务验证流程
