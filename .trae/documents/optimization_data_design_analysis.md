# 策略优化结果数据设计分析

## 当前设计

### 数据关系图

```
┌─────────────────────────────────────────────────────────────────┐
│  PostgreSQL (1条记录)                                           │
│  ├─ task_id: "opt_123"                                          │
│  ├─ strategy_id: "trend_001"                                    │
│  ├─ method: "grid_search"                                       │
│  ├─ target: "sharpe_ratio"                                      │
│  └─ results: [  ← 包含多个结果                                  │
│       {params: {...}, performance: {...}},                     │
│       {params: {...}, performance: {...}},                     │
│       ...                                                      │
│     ]                                                          │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 一对多展开
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  前端显示 (多条记录)                                             │
│  ├─ id: "opt_123_0" (虚拟ID)                                    │
│  ├─ id: "opt_123_1" (虚拟ID)                                    │
│  └─ ...                                                        │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 多对一映射
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  应用操作 (单条记录)                                             │
│  └─ 通过 task_id + index 定位到具体结果                         │
└─────────────────────────────────────────────────────────────────┘
```

## 设计评估

### ✅ 合理之处

| 方面 | 说明 |
|------|------|
 | **存储效率** | 一个任务产生多个结果，存储为一条记录更高效 |
| **数据一致性** | 同一任务的结果放在一起，便于管理和追踪 |
| **查询性能** | 按任务查询比按结果查询更快，减少JOIN操作 |
| **业务逻辑** | 优化任务是一个整体，结果是任务的产出 |

### ⚠️ 潜在问题

| 问题 | 影响 | 解决方案 |
|------|------|----------|
| 虚拟ID | 前端显示的ID在数据库中不存在 | 使用复合ID格式 `task_id_index` |
| 删除复杂 | 删除单条结果需要更新整个记录 | 添加结果级别的状态标记 |
| 应用映射 | 需要解析ID获取task_id和index | 统一ID解析逻辑 |

## 改进建议

### 1. 添加结果级别标识

```json
{
  "task_id": "opt_123",
  "results": [
    {
      "result_id": "opt_123_0",
      "params": {...},
      "performance": {...},
      "applied": false,  // 是否已应用
      "applied_at": null // 应用时间
    }
  ]
}
```

### 2. 优化删除逻辑

- 软删除：标记结果为已删除，不物理删除
- 硬删除：当所有结果都被删除时，删除整个任务记录

### 3. 统一ID处理

```python
def parse_result_id(result_id: str) -> Tuple[str, int]:
    """解析复合ID为task_id和index"""
    if '_' in result_id:
        parts = result_id.rsplit('_', 1)
        if len(parts) == 2 and parts[1].isdigit():
            return parts[0], int(parts[1])
    return result_id, 0
```

## 结论

**当前设计是合理的**，符合业务逻辑和性能要求。

一对多、多对一的关系是正常的数据展开和映射，不是设计缺陷。

建议的改进主要是增强功能（如软删除、应用标记），而非改变核心设计。
