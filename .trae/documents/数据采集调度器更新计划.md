# 数据采集调度器更新计划

## 现状分析

### 当前问题
1. **调度器运行但无任务执行**: 调度器状态 `running: true`，但 `total_tasks: 0`
2. **最后测试时间未更新**: 已启用的2个数据源最后测试时间停留在2月15日
3. **健康检测与测试时间分离**: 健康检测正常但不会更新 `last_test` 字段
4. **缺少自动采集任务调度**: 没有定时为已启用的数据源生成采集任务

### 架构分析
- **调度器**: `UnifiedScheduler` - 统一任务调度器，支持多类型任务
- **工作节点**: `DataCollectorWorker` - 数据采集工作器
- **状态监控**: 已有仪表盘展示调度器状态
- **缺少**: 自动任务生成和最后测试时间更新机制

## 更新目标

1. **自动数据采集任务调度**: 定期为已启用的数据源生成采集任务
2. **最后采集时间自动更新**: 采集成功后自动更新 `last_test` 字段
3. **采集历史记录**: 记录每次采集的详细信息和状态
4. **前端显示优化**: 区分健康检测时间和最后采集时间

## 实施步骤

### 阶段1: 自动任务调度器实现
1. 创建数据采集调度管理器 (`DataCollectionSchedulerManager`)
2. 定时检查已启用的数据源
3. 根据配置生成采集任务并提交到统一调度器
4. 添加采集频率配置（每天/每小时/自定义）

### 阶段2: 最后采集时间更新
1. 在数据采集工作器中添加更新 `last_test` 的逻辑
2. 采集成功后自动更新数据源配置
3. 添加采集状态记录（成功/失败/错误信息）

### 阶段3: 采集历史记录
1. 创建采集历史数据库表
2. 记录每次采集的详细信息（时间、数据源、状态、记录数）
3. 提供采集历史查询API

### 阶段4: 前端仪表盘优化
1. 在数据源配置页面区分显示"健康检测时间"和"最后采集时间"
2. 添加采集历史查看功能
3. 优化调度器状态显示

### 阶段5: 测试验证
1. 端到端功能测试
2. 验证自动任务调度
3. 检查最后采集时间更新
4. 性能和稳定性测试

## 技术要点

### 自动任务调度
- 使用 `asyncio` 或 `threading.Timer` 实现定时任务
- 避免重复提交相同任务
- 考虑数据源采集频率限制（如AKShare 1次/天）

### 最后采集时间更新
- 在 `DataCollectorWorker._complete_task` 中添加更新逻辑
- 通过 `data_source_config_manager` 更新配置
- 确保线程安全

### 数据库表设计
```sql
CREATE TABLE data_collection_history (
    id SERIAL PRIMARY KEY,
    source_id VARCHAR(100) NOT NULL,
    collection_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    status VARCHAR(20) NOT NULL, -- success/failed/error
    records_collected INTEGER,
    error_message TEXT,
    start_time TIMESTAMP,
    end_time TIMESTAMP,
    duration_ms INTEGER
);
```
