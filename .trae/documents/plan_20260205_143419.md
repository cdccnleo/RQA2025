# 数据源配置覆盖问题分析与解决方案

## 问题分析

通过检查当前系统状态，发现以下关键问题：

1. **配置状态不一致**：
   - API返回的数据源状态：只有akshare_stock_a是启用状态，其他都是禁用状态
   - 本地配置文件：多个数据源都是启用状态
   - PostgreSQL数据库：有一条配置记录，最后更新时间为2026-02-05 14:30:35.702965

2. **根本原因**：
   - 容器更新时，本地配置文件被复制到容器中
   - 系统优先从PostgreSQL加载配置，但当PostgreSQL不可用时会回退到文件系统
   - 配置同步机制可能存在问题，导致数据库和文件系统之间的配置不一致

## 解决方案

### 1. 检查配置加载顺序

分析 `config_manager.py` 中的 `load_data_sources()` 函数，确保：
- 优先从PostgreSQL加载配置
- 只有当PostgreSQL不可用时才回退到文件系统
- 当从PostgreSQL加载成功后，同步更新本地文件

### 2. 检查容器更新机制

分析Docker构建过程，确保：
- 容器构建时不会覆盖已有的配置文件
- 使用数据卷持久化配置文件和数据库数据
- 确保容器重启后能正确加载最新的配置

### 3. 验证配置同步机制

检查 `save_data_sources()` 函数，确保：
- 保存配置时同时更新内存、文件系统和PostgreSQL
- 确保配置同步的原子性，避免部分更新
- 添加配置版本控制，防止旧配置覆盖新配置

### 4. 测试验证

1. **记录当前配置**：保存当前API返回的数据源配置状态
2. **更新容器**：运行 `docker-compose -f docker-compose.prod.yml up -d --build app`
3. **验证配置**：检查更新后API返回的数据源配置是否与更新前一致
4. **修复问题**：根据测试结果修复配置覆盖问题

## 预期结果

- 容器更新后数据源配置保持一致
- 系统能够正确处理PostgreSQL连接失败的情况
- 配置同步机制能够确保数据库和文件系统之间的一致性

## 技术要点

- 配置加载顺序的优先级管理
- Docker容器的数据持久化
- 配置同步的原子性和一致性
- 配置版本控制机制
- 错误处理和回退策略
