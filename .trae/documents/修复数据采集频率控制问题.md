## 问题诊断结果

### 🔍 根本原因分析

通过代码分析，我发现了**三个关键问题**导致采集频率控制失效：

#### 问题1: 启动延迟计算错误
```python
# service_scheduler.py:372
if time_since_app_startup < self.startup_delay:
    logger.info(f"应用启动延迟中，还需等待 {self.startup_delay - time_since_app_startup:.1f} 秒")
    await asyncio.sleep(...)
    continue
```
**问题**: 启动延迟只检查一次，然后继续执行调度逻辑，没有正确阻止首次采集。

#### 问题2: 历史采集时间加载问题
```python
# service_scheduler.py:188-195
loaded_times = self.persistence.load_last_collection_times()
if loaded_times:
    self.last_collection_times.update(loaded_times)
else:
    logger.debug("未找到历史采集时间，将使用默认值")
```
**问题**: 如果持久化数据丢失（如容器重建），`last_collection_times` 为空，所有数据源都会被视为"从未采集"，立即触发采集。

#### 问题3: 首次采集判断逻辑缺陷
```python
# service_scheduler.py:438-445
last_time = self.last_collection_times.get(source_id, 0)
time_since_last = current_time - last_time

if time_since_last >= interval_seconds:
    # 触发采集
```
**问题**: 当 `last_time = 0`（从未采集）时，`time_since_last = current_time - 0 = current_time`（很大），必然满足 `>= interval_seconds`，导致立即采集。

### 🛠️ 修复方案

#### 方案A: 修复首次采集判断逻辑（推荐）

修改 `service_scheduler.py` 的 `_scheduler_loop` 方法：

```python
# 检查是否到了采集时间
last_time = self.last_collection_times.get(source_id, 0)

# 修复：如果是首次采集（last_time=0），应该等待一个完整的间隔周期
if last_time == 0:
    logger.info(f"数据源 {source_id} 首次启动，将在下一个采集周期执行（间隔: {interval_seconds:.1f}秒）")
    # 设置 last_time 为当前时间，下次检查时会等待完整间隔
    self.last_collection_times[source_id] = current_time
    continue

time_since_last = current_time - last_time

if time_since_last >= interval_seconds:
    # 触发采集
```

#### 方案B: 增强启动延迟机制

修改启动延迟逻辑，确保在启动延迟期间完全阻止采集：

```python
# 在调度循环开始时增加更强的启动延迟检查
if time_since_app_startup < self.startup_delay:
    logger.info(f"应用启动延迟中，跳过所有数据源检查（还需等待 {self.startup_delay - time_since_app_startup:.1f} 秒）")
    await asyncio.sleep(min(self.check_interval, self.startup_delay - time_since_app_startup))
    continue
```

#### 方案C: 持久化存储改进

确保容器重建时持久化数据不丢失：
- 使用 Docker Volume 持久化配置数据
- 或使用 PostgreSQL 存储采集时间（已部分实现）
