# 修复策略执行列表显示已停止策略问题

## 问题描述
策略执行监控页面仍然显示 `model_strategy_1771503574` 策略，尽管该策略的执行状态为 `stopped`，生命周期不存在。

## 根本原因分析

### 1. 缓存层问题
`get_strategy_execution_status` 函数从 Redis 缓存获取数据时，直接返回缓存数据，没有过滤已停止/已退市的策略。

### 2. 服务层问题
`_is_strategy_retired` 函数依赖 `get_strategy_lifecycle` 检查策略是否已归档。如果策略生命周期不存在（文件已被删除），该函数返回 `False`，导致策略不会被过滤。

### 3. 前端层问题
前端渲染表格时只根据状态过滤器进行过滤，默认显示"全部状态"（包括 `stopped` 状态），没有默认过滤已停止的策略。

## 修复方案

### 任务1: 修复后端服务层过滤逻辑
**文件**: `src/gateway/web/strategy_execution_service.py`

修改 `get_strategy_execution_status` 函数：
1. 添加对 `stopped` 状态策略的过滤
2. 修复 `_is_strategy_retired` 函数，当生命周期不存在但执行状态为 `stopped` 时也视为已停止
3. 在缓存返回前进行过滤

### 任务2: 修复前端默认过滤逻辑
**文件**: `web-static/strategy-execution-monitor.html`

修改 `renderStrategyTable` 函数：
1. 默认过滤掉 `stopped` 状态的策略
2. 添加状态过滤器选项"运行中/暂停"（排除 stopped）
3. 保留"全部状态"选项供用户选择

### 任务3: 添加清理功能
**文件**: `src/gateway/web/strategy_execution_service.py`

添加清理已停止策略的功能：
1. 定期清理已停止超过一定时间的策略
2. 提供手动清理 API

## 验收标准
1. 策略执行监控页面默认不显示 `stopped` 状态的策略
2. 用户可以选择"全部状态"查看包括已停止的策略
3. 已退市（archived）的策略不会出现在执行列表中
