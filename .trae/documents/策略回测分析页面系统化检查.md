## 2. 更新逻辑

### 2.1 前端更新机制
1. **初始化加载**：
   - 页面加载时调用多个数据加载函数：
     - `loadStrategies()`: 加载策略列表
     - `loadBacktestStats()`: 加载回测统计数据
     - `loadPerformanceMetrics()`: 加载性能指标
     - `loadBacktestCharts()`: 加载回测图表数据
     - `initializeBacktestConfig()`: 初始化回测配置

2. **实时更新**：
   - **WebSocket**：实现了回测进度的实时推送
   - **定期轮询**：60秒自动更新统计数据和图表
   - **手动更新**：提供"刷新"按钮，手动触发数据加载

3. **错误处理**：
   - API请求失败时显示错误信息
   - 无数据时显示相应的提示

### 2.2 后端更新机制
1. **事件驱动**：
   - 使用EventBus发布回测相关事件
   - 支持的事件类型：`PARAMETER_OPTIMIZATION_STARTED`、`PARAMETER_OPTIMIZATION_COMPLETED`

2. **WebSocket广播**：
   - 回测状态变化时通过WebSocket广播给所有客户端
   - 支持的消息类型：`backtest_progress`

3. **API响应**：
   - 提供实时的策略状态和回测结果
   - 优先从持久化存储返回数据

## 3. 数据持久化实现

### 3.1 存储机制
- **统一持久化**：使用`unified_persistence.py`中的`get_backtest_persistence()`获取持久化实例
- **存储策略**：优先使用PostgreSQL，失败时降级到文件系统

### 3.2 数据库设计
- **表结构**：通过统一持久化模块管理，支持复杂的回测结果存储
- **主要字段**：
  - `backtest_id`：回测ID，主键
  - `strategy_id`：策略ID
  - `status`：回测状态
  - `start_date`：开始日期
  - `end_date`：结束日期
  - `initial_capital`：初始资金
  - `final_capital`：最终资金
  - `total_return`：总收益率
  - `annualized_return`：年化收益率
  - `sharpe_ratio`：夏普比率
  - `max_drawdown`：最大回撤
  - `win_rate`：胜率
  - `total_trades`：总交易次数
  - `equity_curve`：权益曲线
  - `trades`：交易记录
  - `metrics`：详细指标
  - `created_at`：创建时间

### 3.3 持久化操作
- **保存回测结果**：`save_backtest_result()`
  - 同时保存到PostgreSQL和文件系统
  - 自动处理时间戳格式

- **加载回测结果**：`load_backtest_result()`
  - 优先从PostgreSQL加载，失败则从文件系统加载

- **列出回测结果**：`list_backtest_results()`
  - 支持按策略ID过滤
  - 支持结果数量限制
  - 按创建时间排序

- **更新回测结果**：`update_backtest_result()`
  - 先加载现有结果，再更新字段，最后保存

- **删除回测结果**：`delete_backtest_result()`
  - 同时从PostgreSQL和文件系统删除

## 4. 回测执行流程

### 4.1 回测启动流程
1. **前端请求**：用户点击"开始回测"按钮
2. **API调用**：发送POST请求到`/api/v1/backtest/run`
3. **业务流程编排**：使用BusinessProcessOrchestrator管理回测流程
4. **事件发布**：发布回测开始事件到EventBus
5. **回测执行**：调用回测引擎执行策略回测
6. **WebSocket广播**：实时广播回测进度

### 4.2 回测执行
- **回测引擎**：`BacktestEngine`类，负责实际的回测计算
- **模型分析集成**：可选集成ML层服务进行模型预测分析
- **进度更新**：实时更新回测进度
- **结果计算**：计算各种性能指标

### 4.3 回测完成
1. **结果保存**：将回测结果保存到持久化存储
2. **事件发布**：发布回测完成事件到EventBus
3. **WebSocket广播**：广播回测完成结果
4. **前端更新**：触发前端数据刷新

## 5. 系统架构与数据流

### 5.1 数据流
```
前端请求 → API路由 → 服务层 → 回测引擎 → 数据持久化
    ↑                 ↑         ↓         ↓
    |                 |         |         |
    |←←←←←←←←←←←←←←←←←|←←←←←←←←|←←←←←←←|
    WebSocket/轮询     事件总线    状态更新  回测结果
```

### 5.2 核心组件交互
1. **API层**：处理前端请求，返回数据和状态
2. **服务层**：业务逻辑处理，调用回测引擎
3. **回测引擎**：执行实际的回测计算
4. **持久化层**：数据存储和读取
5. **事件总线**：组件间通信
6. **WebSocket**：实时数据推送

### 5.3 可靠性设计
- **双存储机制**：PostgreSQL + 文件系统
- **多层错误处理**：API → 服务 → 回测引擎
- **冗余更新机制**：WebSocket + 定期轮询
- **事件驱动**：通过EventBus实现松耦合通信
- **降级方案**：各组件都有降级机制，确保系统可用性
