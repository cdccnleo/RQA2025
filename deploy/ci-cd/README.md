# RQA2025 CI/CD 流水线配置指南

## 概述

本文档描述了RQA2025项目的CI/CD（持续集成/持续部署）流水线配置，包括GitHub Actions工作流、环境配置、部署脚本和自动化流程。

## 目录结构

```
deploy/ci-cd/
├── README.md                    # 本文档
├── environments.yml             # 环境配置文件
├── deployment-scripts.yml       # 部署脚本配置
└── .github/workflows/          # GitHub Actions工作流
    ├── ci-cd-pipeline.yml      # 主CI/CD流水线
    └── environments.yml        # 环境保护规则
```

## 快速开始

### 1. 环境要求

- GitHub仓库（支持Actions）
- Kubernetes集群（测试和生产环境）
- Docker注册表（推荐使用GitHub Container Registry）
- 必要的密钥和配置

### 2. 配置密钥

在GitHub仓库的Settings > Secrets and variables > Actions中添加以下密钥：

#### 必需密钥
- `STAGING_KUBECONFIG`: 测试环境kubeconfig文件（base64编码）
- `PRODUCTION_KUBECONFIG`: 生产环境kubeconfig文件（base64编码）

#### 可选密钥
- `SLACK_WEBHOOK_URL`: Slack通知webhook URL
- `SMTP_SERVER`: SMTP服务器地址
- `SMTP_USERNAME`: SMTP用户名
- `SMTP_PASSWORD`: SMTP密码

### 3. 环境配置

在GitHub仓库的Settings > Environments中创建以下环境：

- `staging`: 测试环境
- `production`: 生产环境

## 工作流说明

### 主CI/CD流水线

`ci-cd-pipeline.yml` 定义了完整的CI/CD流程：

#### 触发条件
- 推送到 `main`、`develop`、`feature/*` 分支
- 创建版本标签 `v*`
- 创建Pull Request到 `main`、`develop` 分支
- 手动触发（workflow_dispatch）

#### 工作流程

1. **代码质量检查** (code-quality)
   - 代码格式检查（Black、isort）
   - 代码质量检查（Flake8）
   - 类型检查（MyPy）
   - 安全扫描（Bandit、Safety）

2. **单元测试** (unit-tests)
   - 多Python版本测试（3.9、3.10、3.11）
   - 测试覆盖率报告
   - 并行测试执行

3. **集成测试** (integration-tests)
   - 数据库集成测试
   - 缓存集成测试
   - 服务间通信测试

4. **构建Docker镜像** (build-images)
   - 多服务镜像构建
   - 推送到容器注册表
   - 构建缓存优化

5. **部署到测试环境** (deploy-staging)
   - 自动部署到测试环境
   - 冒烟测试验证
   - 部署状态通知

6. **部署到生产环境** (deploy-production)
   - 生产环境部署前检查
   - 安全部署流程
   - 部署后验证

7. **性能测试** (performance-tests)
   - 负载测试
   - 基准测试
   - 性能报告生成

8. **安全扫描** (security-scan)
   - Trivy漏洞扫描
   - SARIF报告上传
   - 安全合规检查

9. **部署后监控** (post-deployment-monitoring)
   - 服务健康状态检查
   - 部署报告生成
   - 监控配置验证

## 环境配置

### 测试环境 (staging)

- **用途**: 功能测试、集成测试、性能测试
- **部署策略**: 滚动更新
- **资源限制**: 最小1个副本，最大5个副本
- **安全级别**: 基础安全配置
- **监控**: 基础监控和告警

### 生产环境 (production)

- **用途**: 生产服务
- **部署策略**: 蓝绿部署、金丝雀部署、滚动更新
- **资源限制**: 最小3个副本，最大15个副本
- **安全级别**: 高级安全配置
- **监控**: 完整监控、告警和日志聚合

## 部署策略

### 蓝绿部署

- **特点**: 零停机时间部署
- **适用场景**: 生产环境主要版本更新
- **配置**: 自动流量切换，健康检查验证

### 金丝雀部署

- **特点**: 渐进式流量切换
- **适用场景**: 新功能验证、风险控制
- **配置**: 10%流量逐步增加到100%

### 滚动更新

- **特点**: 传统滚动更新
- **适用场景**: 小版本更新、热修复
- **配置**: 最大不可用副本数为0

## 自动化流程

### 预部署检查

- 代码质量检查
- 单元测试（覆盖率>80%）
- 集成测试
- 安全扫描
- 依赖检查
- 许可证检查

### 部署流程

- 环境准备
- 配置验证
- 镜像构建
- Kubernetes部署
- 服务验证
- 性能测试（可选）

### 后部署流程

- 健康状态检查
- 功能测试
- 监控配置
- 日志聚合
- 备份策略

## 回滚策略

### 自动回滚

**触发条件**:
- 健康检查失败（3次/5分钟）
- 错误率过高（>5%/5分钟）
- 响应时间过长（>2秒/5分钟）
- 资源使用率过高（>90%/5分钟）

**回滚配置**:
- 最大回滚版本数: 5
- 回滚超时时间: 10分钟
- 保留数据: 是
- 清理资源: 否

### 手动回滚

- 用户触发（需要审批）
- 定时回滚（每天凌晨2点）

## 监控和告警

### 部署监控指标

- 部署持续时间
- 部署成功率
- 回滚率
- 服务运行时间
- 响应时间
- 错误率

### 资源监控指标

- CPU使用率
- 内存使用率
- 磁盘使用率
- 网络I/O

### 告警规则

- **部署失败**: 成功率<95%（严重）
- **回滚频率过高**: 回滚率>10%（警告）
- **服务响应时间过长**: 响应时间>2秒（警告）
- **资源使用率过高**: CPU/内存>90%（警告）
- **磁盘空间不足**: 使用率>85%（严重）

## 故障排除

### 常见问题

1. **部署失败**
   - 检查Kubernetes集群状态
   - 验证镜像是否成功构建
   - 检查资源配置和限制

2. **测试失败**
   - 检查测试环境配置
   - 验证依赖服务状态
   - 查看测试日志

3. **性能问题**
   - 检查资源使用情况
   - 分析性能测试报告
   - 优化资源配置

### 调试步骤

1. 查看GitHub Actions日志
2. 检查Kubernetes事件和日志
3. 验证服务健康状态
4. 检查监控指标和告警

## 最佳实践

### 开发流程

1. 在功能分支开发新功能
2. 创建Pull Request到develop分支
3. 通过CI检查后合并到develop
4. 自动部署到测试环境
5. 测试验证通过后合并到main分支
6. 自动部署到生产环境

### 部署建议

1. 使用语义化版本标签
2. 在生产环境部署前充分测试
3. 监控部署过程和后续状态
4. 准备回滚计划

### 安全考虑

1. 定期更新依赖包
2. 扫描安全漏洞
3. 使用最小权限原则
4. 加密敏感信息
