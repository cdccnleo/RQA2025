# 启动脚本导入卡住问题诊断指南

## 诊断工具

已创建以下诊断工具来定位启动脚本在导入应用时卡住的问题：

### 1. 增强的启动脚本 (`scripts/start_api_server.py`)

**新增功能**：
- ✅ 时间戳记录：每个步骤都记录执行时间
- ✅ 日志刷新机制：确保日志及时输出（禁用缓冲）
- ✅ 超时检测：60秒超时警告
- ✅ 详细日志：在关键步骤（导入 app_factory）前后添加详细诊断信息

**关键改进**：
- 使用 `sys.stdout.flush()` 和 `sys.stderr.flush()` 强制刷新输出
- 使用 `time.time()` 记录每个步骤的耗时
- 在导入 `app_factory` 时分为3个子步骤记录时间

### 2. 诊断脚本 (`scripts/diagnose_import.py`)

**功能**：
- 逐步导入各个模块，记录每个步骤的时间
- 检测是否有阻塞操作
- 输出详细的诊断报告

## 使用方法

### 方法1：使用增强的启动脚本

直接在容器中运行增强的启动脚本：

```bash
# 在容器中运行
docker exec -it rqa2025-rqa2025-app-1 python scripts/start_api_server.py
```

**观察要点**：
- 查看每个步骤的耗时
- 特别关注 "🔍 关键步骤：开始导入主应用（app_factory）" 部分
- 如果某个步骤耗时超过5秒，会显示警告
- 如果超过60秒，超时检测线程会发出警告

### 方法2：使用诊断脚本

运行独立的诊断脚本：

```bash
# 在容器中运行
docker exec -it rqa2025-rqa2025-app-1 python scripts/diagnose_import.py
```

**输出说明**：
- 每个模块导入的耗时
- 如果某个模块导入耗时超过5秒，会显示警告
- 最后会测试创建应用的过程

## 诊断步骤

### 步骤1：运行诊断脚本

```bash
docker exec -it rqa2025-rqa2025-app-1 python scripts/diagnose_import.py
```

### 步骤2：分析输出

查看输出中的时间戳和耗时：
- 如果某个步骤耗时异常长（>5秒），说明该步骤可能包含阻塞操作
- 如果脚本在某个步骤后停止，说明该步骤卡住了

### 步骤3：检查具体模块

根据诊断脚本的输出，检查耗时最长的模块：
- 如果是 `DataManager`，检查其初始化过程
- 如果是 `api` 模块，检查模块级别的代码
- 如果是 `app_factory`，检查其导入的依赖

## 可能的问题和解决方案

### 问题1：日志缓冲导致消息延迟

**症状**：日志消息没有及时输出

**解决方案**：已解决
- 启动脚本已禁用输出缓冲
- 使用 `sys.stdout.flush()` 强制刷新

### 问题2：模块导入时执行阻塞操作

**症状**：某个模块导入耗时很长

**解决方案**：
- 将阻塞操作移到 `lifespan` 或路由处理中
- 使用异步方式执行阻塞操作
- 添加超时机制

### 问题3：数据库连接超时

**症状**：在导入 `DataManager` 或相关模块时卡住

**解决方案**：
- 检查数据库连接配置
- 确保数据库服务已启动
- 添加连接超时设置

### 问题4：循环依赖或死锁

**症状**：导入过程完全停止，无任何输出

**解决方案**：
- 检查模块间的导入关系
- 使用延迟导入（lazy import）
- 重构模块结构，消除循环依赖

## 预期结果

运行诊断脚本后，应该能看到：

1. **每个模块的导入时间**：正常情况下每个模块导入应该在1-2秒内完成
2. **总耗时**：正常情况下总耗时应该在10-20秒内
3. **警告信息**：如果某个步骤耗时超过5秒，会显示警告

## 下一步

根据诊断脚本的输出：
1. 如果定位到具体卡住的模块，检查该模块的初始化代码
2. 如果发现阻塞操作，将其移到异步初始化中
3. 如果发现超时问题，添加超时机制或检查依赖服务
